**directives/onboard-engineer.md** - Add 2 Known Issues:

1. **KI-001: Python pytest imports fail with ModuleNotFoundError**
   - Discovered: 2026-02-16 during Phase 6 Plan 02
   - Symptom: Running `pytest tests/execution/` fails with `ModuleNotFoundError: No module named 'execution'`
   - Root cause: Pytest's module collection phase runs before conftest.py executes, so sys.path manipulation in conftest is too late. The `execution/` directory is not a standard Python package on the import path.
   - Resolution: Created setup.py with setuptools configuration and ran `pip install -e .` to install the execution package in editable mode. Also added `pythonpath = .` to pytest.ini as a belt-and-suspenders fix.
   - Prevention: Always run `pip install -e .` after cloning the repo (documented in Step 4 of onboarding procedure). If you see ModuleNotFoundError for any `execution.*` import, re-run the editable install.

2. **KI-002: validate_env.py --skip flag order matters on some Python versions**
   - Discovered: 2026-02-16 during Phase 6 Plan 02
   - Symptom: Running `python execution/validate_env.py --skip secrets --skip firestore` may produce unexpected behavior if argparse nargs configuration changes
   - Root cause: Multiple --skip flags use `append` action in argparse; if the default list is non-empty, flags append to it rather than replacing
   - Resolution: Ensure --skip default is an empty list `[]` in argparse configuration
   - Prevention: When adding new --skip options, test with both single and multiple flags

**directives/deploy.md** - Add 1 Known Issue:

1. **KI-001: Docker multi-stage Python venv portability between Debian and Alpine**
   - Discovered: 2026-02-16 during Phase 6 Plan 03
   - Symptom: Python packages with C extensions (e.g., grpcio) compiled in python:3.12-slim (Debian) may segfault when copied to node:20-alpine (Alpine/musl)
   - Root cause: Debian uses glibc while Alpine uses musl libc. Native extensions compiled against glibc are not binary-compatible with musl.
   - Resolution: Current requirements.txt uses pure-Python packages or packages with pre-built musl wheels. If adding new Python dependencies with C extensions (numpy, grpcio, etc.), must either: (a) install from Alpine-compatible wheels, (b) compile in Alpine stage, or (c) add `apk add` for build dependencies in the runner stage.
   - Prevention: Before adding new Python dependencies to requirements.txt, check if they have C extensions. Test the Docker build end-to-end with `docker build .` locally before merging.

**directives/generate-embeddings.md** - Add 1 Known Issue:

   - Discovered: 2026-02-16 during Phase 6 Plan 02
   - Symptom: 15 of 65 pytest tests fail with assertion errors on expected error message strings (e.g., expecting "Firestore accessible" but getting "Firestore connected")

  <action>
Create a CLI tool `execution/log_incident.py` that automates the self-annealing loop: when an incident occurs, the engineer runs this script to append a properly formatted Known Issues entry to the relevant directive.

**execution/log_incident.py** requirements (target: 80-120 lines):

1. Uses argparse with these required arguments:
   - `--directive` (required): Name of the directive file (e.g., "deploy" or "deploy.md" -- handle both)
   - `--title` (required): Short title for the incident
   - `--symptom` (required): What was observed
   - `--cause` (required): Root cause
   - `--resolution` (required): How it was fixed
   - `--prevention` (required): How to avoid in future

2. Logic:
   - Resolve directive path: `directives/{name}.md` (append .md if not present)
   - Validate the directive file exists, exit 1 with clear error if not
   - Read the directive file content
   - Find the "## Known Issues" section
   - Count existing KI-NNN entries to determine next number (parse `### KI-` prefixes)
   - Generate next KI number (KI-001 if none exist, KI-003 if KI-001 and KI-002 exist, etc.)
   - Format the entry using the standard template:
     ```
     ### KI-{NNN}: {title}
     **Discovered:** {today's date YYYY-MM-DD}
     **Symptom:** {symptom}
     **Root cause:** {cause}
     **Resolution:** {resolution}
     **Prevention:** {prevention}
     ```
   - Insert the new entry BEFORE the "## Post-Operation" section (or at end of Known Issues section if Post-Operation not found)
   - Write the updated file
   - Print confirmation: "Added KI-{NNN} to directives/{name}.md"
   - Exit 0 on success

3. Add `--dry-run` flag that prints what would be added without writing the file.

4. Include `if __name__ == "__main__":` guard calling main().

**tests/execution/test_log_incident.py** requirements (target: 50-80 lines):

1. Use pytest and tmp_path fixture to create temporary directive files
2. Test cases:
   - test_log_incident_creates_first_entry: Empty Known Issues -> KI-001 created
   - test_log_incident_increments_number: Existing KI-001 -> KI-002 created
   - test_log_incident_dry_run: File not modified with --dry-run flag
   - test_log_incident_missing_directive: Non-existent file -> exit 1
   - test_log_incident_appends_before_post_operation: Entry inserted before "## Post-Operation"

3. Create a helper function that generates a minimal directive fixture:
   ```
   ## Known Issues

   {existing_content}

   ## Post-Operation
   ```

4. Use subprocess.run or import the main function directly to test.

Do NOT add log_incident.py to requirements.txt (it has no external dependencies beyond Python stdlib).
  </action>
