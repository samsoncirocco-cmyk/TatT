**Create `firestore.rules` in project root:**

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // User designs — owner access only
    match /users/{userId}/designs/{designId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;

      // Versions subcollection
      match /versions/{versionId} {
        allow read, write: if request.auth != null && request.auth.uid == userId;

        // Layers subcollection
        match /layers/{layerId} {
          allow read, write: if request.auth != null && request.auth.uid == userId;
        }
      }
    }

    // User preferences — owner access only
    match /users/{userId}/preferences/{document} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }

    // Deny all other paths by default (Firestore denies by default, but explicit is better)
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

Key decisions:
- No public read on designs (private by default)
- Auth required for ALL operations (request.auth != null)
- User can only access their own path (request.auth.uid == userId)
- Subcollections inherit the userId check pattern
- No admin bypass in rules (firebase-admin SDK bypasses rules server-side)
- The catch-all deny at bottom is defense-in-depth

**Create `storage.rules` in project root:**

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {

    // Generated images: public read (for sharing), owner write only
    match /generated/{userId}/{designId}/{versionId}/{fileName} {
      allow read: if true;
      allow write: if request.auth != null && request.auth.uid == userId;
      allow delete: if request.auth != null && request.auth.uid == userId;
    }

    // Deny all other paths
    match /{allPaths=**} {
      allow read, write: if false;
    }
  }
}
```

Key decisions:
- Generated images are publicly readable (enables sharing links, no auth needed to view)
- Only the owning user can write/delete their images
- All other paths denied
- File size limits can be added later if needed (e.g., `request.resource.size < 10 * 1024 * 1024` for 10MB max)

  <action>
**Create `src/services/storage/migrationService.ts`:**

This service runs once when an anonymous user authenticates (signs up or logs in), migrating their localStorage data to Firestore.

**Functions:**

1. `hasPendingMigration(): boolean`
   - Checks if localStorage has any design/version data that hasn't been migrated
   - Scans for keys matching: `design_anonymous_*`, `versions_anonymous_*`, `tattester_version_history_*`, `canvas_layers` (the Zustand persist key)
   - Returns true if any data found
   - This is a quick synchronous check — no Firestore calls

2. `migrateLocalStorageToFirestore(userId: string): Promise<MigrationResult>`
   - Return type: `{ designsMigrated: number, versionsMigrated: number, layersMigrated: number, errors: string[] }`
   - Steps:
     a. Create FirestoreAdapter instance
     b. Scan localStorage for all design-related keys:
        - `tattester_version_history_*` — old versionService format (sessionId-based)
        - `design_anonymous_*` — if LocalStorageAdapter was used anonymously
        - `canvas_layers` — current Zustand persisted state
     c. For each sessionId with version history:
        - Generate a new designId (UUID)
        - Create a Design document in Firestore with metadata
        - For each version: call `storage.saveVersion()` and `storage.saveLayers()`
        - Track counts for result
     d. For Zustand canvas_layers:
        - Parse the persisted state
        - If layers exist, create a new design + version with those layers
     e. After ALL data successfully written to Firestore, mark migration as complete:
        - Set `localStorage.setItem('migration_completed_${userId}', new Date().toISOString())`
     f. Do NOT delete localStorage data yet (safety net — user can fall back)
     g. Return MigrationResult

3. `clearMigratedLocalStorage(userId: string): void`
   - Called manually or after user confirms migration success
   - Removes all localStorage keys that were migrated
   - Only runs if `migration_completed_${userId}` exists
   - This is the "point of no return" — separate from migration for safety

4. `isMigrationComplete(userId: string): boolean`
   - Checks if `migration_completed_${userId}` exists in localStorage
   - Used to skip migration on subsequent loads

**Idempotency:**
- `migrateLocalStorageToFirestore` checks `isMigrationComplete()` first — returns early if already done
- If migration fails partway, no completion marker is set, so next call retries
- Firestore `setDoc` with merge is idempotent — writing same data twice is safe

**Error handling:**
- Wrap each design migration in try-catch
- On per-design failure: log error, add to errors array, continue with next design
- On complete failure: throw with all accumulated errors
- Never set completion marker if any errors occurred
