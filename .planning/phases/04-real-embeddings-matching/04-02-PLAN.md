Create `src/services/firestore-vector-service.ts` to replace `src/services/vectorDbService.ts` (Supabase-based).

Imports:
- `getFirestore`, `FieldValue` from `firebase-admin/firestore`
- `normalizeVector` from `../lib/embedding-normalization`

Collection: `artist_embeddings` (text embeddings, 768 dimensions from text-embedding-004).

Export `storeArtistEmbedding(artistId: string, embedding: number[], metadata?: { sourceText?: string, model?: string }): Promise<void>`:
- Normalize embedding with `normalizeVector()` before storing
- Store in `artist_embeddings/{artistId}` document with fields:
  - `embedding`: `FieldValue.vector(normalized)` — Firestore native vector field
  - `dimension`: 768
  - `model`: metadata.model or `'text-embedding-004'`
  - `sourceText`: metadata.sourceText (the text that was embedded, for debugging)
  - `createdAt`: `FieldValue.serverTimestamp()`

Export `searchSimilarArtists(queryEmbedding: number[], limit?: number): Promise<Array<{ artistId: string; score: number; rank: number }>>`:
- Normalize queryEmbedding with `normalizeVector()`
- Use Firestore `findNearest()` API:
  ```
  const results = await db.collection('artist_embeddings')
    .findNearest({
      vectorField: 'embedding',
      queryVector: normalized,
      distanceMeasure: 'COSINE',
      limit: limit || 20
    })
    .get();
  ```
- Map results: `{ artistId: doc.id, score: 1 - doc.get('distance'), rank: index + 1 }`
  - Note: Firestore COSINE distance returns 0 for identical, 2 for opposite. Convert: `similarity = 1 - (distance / 2)`.
  - Check actual Firestore findNearest response format — the distance may be a field on the snapshot or accessed differently. If `doc.get('distance')` doesn't work, check `doc.distance` or the query result metadata.
- Default limit: 20

Export `getArtistEmbedding(artistId: string): Promise<{ embedding: number[]; model: string } | null>`:
- Read from `artist_embeddings/{artistId}`
- Return null if document doesn't exist

Do NOT modify or delete `vectorDbService.ts` — it will be swapped out in Plan 03 when hybridMatchService is rewired.

Create `src/services/match-config-service.ts`:

Export interface `MatchConfig`:
```typescript
export interface MatchConfig {
  rrfWeights: {
    graph: number;  // 0-1, weight for Neo4j graph results
    vector: number; // 0-1, weight for vector similarity results
  };
  rrfK: number;              // RRF K parameter (default 60)
  confidenceThreshold: number; // Minimum score to include in results (default 0.5)
}
```

Export `DEFAULT_MATCH_CONFIG: MatchConfig` constant:
```typescript
export const DEFAULT_MATCH_CONFIG: MatchConfig = {
  rrfWeights: { graph: 0.5, vector: 0.5 },
  rrfK: 60,
  confidenceThreshold: 0.5
};
```

Export `getMatchConfig(): Promise<MatchConfig>`:
- Read from Firestore: `db.collection('config').doc('matching').get()`
- If document doesn't exist, return `DEFAULT_MATCH_CONFIG`
- If document exists, merge with defaults (so partial config documents work): `{ ...DEFAULT_MATCH_CONFIG, ...configDoc.data() }`
- Validate: ensure rrfWeights.graph + rrfWeights.vector > 0 (prevent zero-weight edge case). If invalid, log warning and return defaults.
- Add 60-second in-memory cache to avoid reading Firestore on every match request. Cache structure: `{ data: MatchConfig, timestamp: number }`. TTL: 60s. This is NOT the query embedding cache — this is config-only.

Export `weightedRRF(graphResults: Array<{id: string; rank: number}>, vectorResults: Array<{id: string; rank: number}>, config: MatchConfig): Map<string, number>`:
- Implement weighted RRF: for each result list, `score = weight * (1 / (k + rank))`
- Merge scores by artist ID
- Return Map of artistId -> combined RRF score
- This replaces the hardcoded `reciprocalRankFusion` function in `matchService.ts`
